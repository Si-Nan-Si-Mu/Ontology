## 2. 项目背景：要解决的问题

### 2.1 行业痛点

- **大模型直接「猜人格」**：输出难复现、难合并多源材料、**证据链弱**，不利于审计与合规产品化。  
- **人格相关数据分散**：聊天、自述、量表、行为日志各自为政，缺少统一「自我—情境—关系—事件」的**可查询结构**。  
- **心理科技与数字化场景**需要：在**不冒充临床诊断**的前提下，对沟通风格、压力情境、关系模式等做**结构化整理与可追溯推断**。

### 2.2 POG 的切入方式

用 **信息科学意义上的本体（Ontology）+ 图数据库（Neo4j）** 把「人格相关证据与假设」落成**可版本化、可查询、可合并增量**的数据层；生成式 AI（如 DeepSeek）负责**抽取与表达增强**，**图与规则**负责边界与一致性——形成 **「生成—约束—反馈」** 闭环。

## 3. 项目目的与价值主张

### 3.1 核心目的

1. **建一层「人格—社会情境」基层数据**：以 `Person` 为锚点，将可验证材料写入图库，支持后续查询、导出与推理。  
2. **可解释**：推断结果尽量以**节点 / 关系 / 批次**形式存在，便于回答「依据是什么」。  
3. **可演进**：本体定义（[`backend/ONTOLOGY.yaml`](../backend/ONTOLOGY.yaml)）与写入代码可迭代，同一 `subject_id` 上支持新批次覆盖旧人格子图等业务策略。

### 3.2 价值主张（一句话）

> **把「关于我」的多源材料，变成 Neo4j 里可查询、可审计、可迭代的人格本体图；AI 辅助抽取与叙述，图库承载事实与版本。**

### 3.3 不是什么（边界声明）

- **非**临床心理诊断、**非**标准化职业测评工具。  
- **非**对「他人」未经授权的画像产品；展演与开发须使用**虚构数据或本人已授权数据**。  
- 详见 [`ETHICS_AND_DATA.md`](ETHICS_AND_DATA.md)、[`PERSONA_INFERENCE_REFERENCES.md`](PERSONA_INFERENCE_REFERENCES.md)。

### 3.4 跨领域效用：信用体系、心理干预、社会工程（创新与边界）

以下从**「结构化 + 可审计 + 可版本化」**这一共同底座出发，说明 POG 在相关议题上的**可能创新点与实际效用**；路演时可各用 1～2 句带过，并**必须**与 §3.3、§9 及 [`ETHICS_AND_DATA.md`](ETHICS_AND_DATA.md) 的禁止性条款一并陈述。

#### 信用体系（声誉、合规与可解释治理）

- **创新点**：将「关于某人的多源材料与推断」从不可见的模型上下文，提升为 **Neo4j 中的节点—关系—批次—证据引用**；在组织治理、合规审计、研究复现等场景中，可回答「结论依附于哪些可核验片段、哪次导入、哪版抽取器」，降低**隐性声誉标签**与**口头叙事**的信息不对称。  
- **效用（合规取向）**：作为**人机协同的附件层**——例如内部调查、客服升级、学术纵向研究中的**材料编排与假设管理**；强调**补充性透明度**，而非替代法定征信或评分模型。  
- **边界**：**禁止**将图谱推断作为信贷、保险核保、招聘筛选等高风险自动化决策的**唯一或决定性依据**（见伦理文档 §2.2）；若涉及自动化决策，须符合当地法律与解释义务，并保留人工复核与异议渠道。

#### 心理干预与心理科技（辅助而非治疗）

- **创新点**：在**不冒充临床诊断**的前提下，把聊天记录、自述、会谈摘要等整理为 **「观测层（交互/话语）—推断层（TraitHypothesis）—证据链」** 的可查询结构，支持个案概念化、会谈前后对齐、多来源材料合并，减少「模型一次性总结、无法回溯」的协作成本。  
- **效用**：EAP、教练、教育场景中**提升觉察与沟通效率**的**工作台数据层**；未来与图约束叙述（[`TECH_SPEC.md`](TECH_SPEC.md) §5.9）结合后，可生成**带引用路径**的反思性文本，仍须标注不确定性。  
- **边界**：**非**心理治疗、**非**精神障碍诊断工具；涉及来访者数据须执业规范、知情同意与最小必要原则；严重风险须转介专业机构。

#### 社会工程与安全意识（防御与沟通仿真）

- **创新点**：在**授权红队 / 安全意识教育**中，将钓鱼与社工剧本中的**角色关系、话术压力点、信任建立序列**建模为可查询子图，便于对比「正常沟通」与「高压诱导」在结构上的差异，支撑**可重复演练与复盘**。  
- **效用**：企业**社工防御**演练的数据层、客服/管理中的**冲突降级与谈判沟通**结构化复盘（与 [`PROJECT_BRIEF.md`](PROJECT_BRIEF.md) §5.3 表述一致）。  
- **边界**：**禁止**未授权渗透、欺诈话术生成、对他人画像以实施攻击；对外材料统一使用「社工防御」「安全意识教育」「互动沟通仿真」等中性表述。

## 4. 项目内容：POG 是什么

### 4.1 名称与定位

- **POG**：Personality Ontology Graph（人格本体图），工作用产品名；品牌与商标策略见 [`BRAND_AND_TRADEMARK.md`](BRAND_AND_TRADEMARK.md)。  
- **定位**：心理学 × 知识图谱 × 生成式 AI 方向的 **可演进基层数据与推理层原型**（不是单一黑盒聊天机器人）。

### 4.2 三层语义（避免「本体论」歧义）

- **信息本体（TBox）**：类、属性、关系、约束——当前以 `ONTOLOGY.yaml` + Neo4j 标签/关系类型落地。  
- **实例数据（ABox）**：具体 `Person`、特质节点、典型对话片段等。  
- **运行库**：Neo4j 5.x；API 为 FastAPI；前端 Vue 3 + Vite。

### 4.3 当前数据形态（MVP）

- 以 **微信 / 聊天 JSON** 为主入口：解析后经（可选）DeepSeek 增强，写入 **Person + 外向特质子图**（摘要、表达风格、口癖、情绪维度、社会关系草图、大五/MBTI/九型占位或假设、典型对话、分析元数据、显著特质等），关系类型以 `HAS_*` 为主。  
- **不落库逐条 Utterance** 全量聊天行：控制隐私面与存储；人格侧以**聚合节点 + 典型片段**为主。

## 5. 差异化：与「纯 LLM 人格 Bot」对比

| 维度 | 纯 LLM 端到端 | POG 路线 |
| --- | --- | --- |
| 事实存放 | 隐在模型权重或单次上下文 | **Neo4j 显式节点与边**，可查询、可导出 |
| 多源合并 | 难、易冲突 | **同一 `subject_id` 锚点**上批次化写入与覆盖策略 |
| 可审计性 | 弱 | **批次 ID、导出包、子图 API** |
| AI 角色 | 「全知叙述」 | **抽取 / 摘要 / 未来：图约束叙述**（见 §8.1 第一层） |

## 6. 当前已实现能力（展演可点功能）

## 7. 推荐演示脚本

### 7.1 环境与数据（演示前 5 分钟）

1. Neo4j 可用，`GET /health/neo4j` 返回 `ok: true`。  
2. 后端 `uvicorn`、前端 `npm run dev` 已启动（见 [`OPERATIONS.md`](OPERATIONS.md)）。  
3. 准备 **虚构** 聊天 JSON 或脱敏样例；**禁止**在公开展示中使用他人真实聊天记录。

### 7.2 约 10 分钟口播 + 操作线

| 时间 | 动作 | 讲解要点 |
| --- | --- | --- |
| 0:00–1:00 | 封面 / 一句话 | §3.2 价值主张 + §3.3 边界 |
| 1:00–3:00 | 打开「数据导入」 | 来源选「聊天 JSON」；上传或（可选）wx-cli 选会话；**本体 Person / 客体 sender** 与 `subject_id` 含义 |
| 3:00–5:00 | 点击「导入聊天 JSON」 | 写入的是**图上的结构化子图**，不是聊天机器人记忆 |
| 5:00–7:00 | 「图数据」页选 Person | 子图可视化；外向 `HAS_*` 关系 |
| 7:00–9:00 | 「开放接口」或 Swagger | `persona-export` 下载 JSON/文本；**可审计、可带走** |
| 9:00–10:00 | 收尾 | §8 未来方向 + §9 伦理一句 |

### 7.3 约 20 分钟（加深度）

在 10 分钟版基础上增加：

- **OpenAPI**：展示请求体字段与错误码语义。  
- **`ONTOLOGY.yaml`**：打开 1～2 个类定义，说明「语义真源」与图标签的对应。  
- **伦理与文献**：[`ETHICS_AND_DATA.md`](ETHICS_AND_DATA.md)、[`PERSONA_INFERENCE_REFERENCES.md`](PERSONA_INFERENCE_REFERENCES.md) 各 1 页。

## 8. 未来方向

### 8.1 技术路线（与 `PROJECT_BRIEF` §6.1 对齐）

| 层级 | 方向 | 说明 |
| --- | --- | --- |
| **第一层（近期）** | **图约束的 DeepSeek / LLM** | 以 Neo4j 中 **Person + 已入库特质子图 + 结构化证据** 为**唯一事实底层**做问答与解释；详见 [`TECH_SPEC.md`](TECH_SPEC.md) **§5.9** 与规划中的 **`POST /narrate`**。导入管线中 LLM 与「按库回答」将逐步统一到**可引用子图**的协议上。 |
| **第二层** | 多档案、多情境推演 | 跨 Person、反事实边编辑、干预路径仿真；仍以图变更可审计为前提。 |
| **第三层** | 混合检索与量表 | 向量检索、外部量表、多模态证据与图谱的联合置信度。 |

### 8.2 产品与商业（概要）

- **B2B API**：图谱写入、子图查询、导出、（未来）图约束叙述接口。  
- **垂直场景**：EAP/教练工作台材料结构化、企业**社工防御**演练数据层、游戏 NPC 关系记忆等（合规表述见 [`PROJECT_BRIEF.md`](PROJECT_BRIEF.md)）。  
- **计费锚点**（规划）：活跃档案数、API 调用量、存储、私有化部署。

### 8.3 工程与数据治理

- 预置 **Cypher 模板查询**（防注入）、租户隔离、审计日志。  
- 任务队列承载长耗时导入；可选对象存储保留原始文件副本（策略待定）。  
- 版本号与 schema 迁移（如 `facet_schema_version`）持续与 `ONTOLOGY.yaml` 对齐。

## 9. 伦理与合规（展演必提）

建议在演示 **开场或收尾各强调一次**：

1. 输出为 **探索性假设**，不替代临床或人事决策。  
2. 仅处理 **本人授权或虚构数据**；禁止未授权他人画像与攻击性用途。  
3. 区分 **观测层（可溯源材料）** 与 **推断层（假设节点）**；对外交付须带免责声明与引用路径。

## 11. 备用 Q&A（简答）

**Q：和「大模型套壳人格助手」有何不同？**  
A：人格结论与证据在**图里**；模型是管道的一环，且未来将**按库约束**输出，而不是不可审计的黑盒。

**Q：能诊断抑郁症 / MBTI 吗？**  
A：**不能也不应**作临床诊断；MBTI 等在文献中单模态证据弱，仅作占位或弱假设并带局限说明。

**Q：数据存在哪？**  
A：结构化图数据在 **Neo4j**；密钥在 `.env`，勿提交仓库。

**Q：开源与商业？**  
A：代码与文档可开源协作；商业命名、商标与 VI 见品牌文档。
